# 한국어·영어 프롬프트 인젝션 퍼징 통합 기준서

부제: **`1차자료.md` + `2차 자료.md` 종합판**  
목표: 공통 유형은 결합하고, 상이 유형은 추가하여 **한국어/영어 프롬프트 인젝션 퍼징용 단일 기준 문서**를 만든다.

---

## 0. 문서 목적

이 문서는 사용자가 조사한 두 자료를 하나로 합쳐 다음을 동시에 만족하도록 재구성한 통합본이다.

1. **공통 유형 병합**: 두 자료에서 사실상 같은 의미로 다뤄진 유형은 하나의 통합 패밀리로 묶는다.
2. **상이 유형 추가**: 한 자료에만 명시된 유형은 별도 유형으로 추가한다.
3. **영어/한국어 분리 정리**: 영어 퍼징 유형과 한국어 퍼징 유형을 각각 정리하되, 비교 가능한 경우는 같은 공통 IR로 연결한다.
4. **근거·출처 추적성 확보**: 각 유형마다 직접 근거와 보강 출처를 명시한다.
5. **실전용 퍼징 자료화**: 실제 악성 원문 나열보다 **slot 기반 템플릿 + mutation family + carrier context + split 정책** 중심으로 정리한다.

> 주의
> - 이 문서는 **보안 검증용/레드팀용 퍼징 taxonomy**를 만드는 것이 목적이다.
> - 실제 공격 문장을 대량으로 싣기보다 **재현 가능한 템플릿**과 **변이 규칙**을 우선한다.
> - 내부 PII/Guardrail 문서들은 **직접적인 prompt injection benchmark가 아니라**, 한국어 고유 변이 우선순위와 EN-KO 격차를 설명하는 **transfer evidence**로만 사용한다.

## 2. 병합 원칙

1. **같은 목적, 같은 공격면, 같은 효과**를 가진 유형은 하나로 합친다.
2. **표면은 비슷해도 공격면이 다르면 분리**한다. 예를 들어 일반 indirect와 email-agent, repo/coding-agent, tool-agent는 분리한다.
3. **한국어는 영어 번역판으로 대체하지 않는다.** EN-KO paired base는 유지하되, 한국어 고유 mutation layer를 독립시킨다.
4. **유형(type)** 과 **변이 패밀리(family)** 를 분리한다. 예를 들어 `EN-02 indirect_document_rag`는 유형이고, `B01 구조적 래퍼`, `K04 자모/유사문자`, `A04 긴박감`은 결합 가능한 변이 패밀리다.
5. **평가용 split discipline** 을 반드시 포함한다. `public_train / external_heldout / private_adaptive / benign_hard_negative / bootstrap_only`를 분리한다.

## 4. 공통 분류축 (공통 IR)

| 축 | 설명 | 왜 필요한가 |
|---|---|---|
| **attack_surface** | direct_user / indirect_document / email / repo / tool_agent / multimodal | 영어·한국어를 같은 프레임에서 비교하기 위해 |
| **goal** | override / prompt_leak / output_steering / unauthorized_tool_use / pii_exfiltration / over_defense | 같은 표면이라도 목표가 다르면 다른 유형으로 봐야 하기 때문 |
| **carrier_context** | plain chat / mail / web / markdown / HTML / JSON / YAML / table / repo / RAG_chunk / tool output | 실제 앱 공격면 다양성을 보존하기 위해 |
| **mutation_family** | direct override / wrapper / obfuscation / jamo / choseong / kr_digits / code_switch / style / urgency / footer / separator | 성공/실패 원인을 mutation 단위로 추적하기 위해 |
| **language_scope** | en_only / ko_only / en_ko_paired / multilingual | 언어 효과와 공격 난이도를 분리하기 위해 |
| **evaluation_split** | public_train / external_heldout / private_adaptive / benign_hard_negative / bootstrap_only | 과적합과 held-out 오염을 막기 위해 |
| **recommended_layer** | Layer 1 / Layer 2 / Layer 3 / Layer 4 | 어느 수준에서 검증해야 하는지 명시하기 위해 |

## 6. 영어 프롬프트 인젝션 퍼징 유형

### EN-01. 직접 사용자 인젝션 (direct_user)

**정의**  
사용자가 직접 모델 입력창에 상위 지시 무시, 숨은 규칙 노출, 특정 출력 형식 강제 등을 요구하는 유형.

**대표 attack surface**  
plain chat, pseudo-system block, fake developer note, delimiter break, quoted instruction.

**권장 EN slot template**
- `[priority-override phrase] + [new top-level rule] + [forced output format]`
- `[policy-bypass framing] + [request for hidden rule or internal behavior]`
- `[role-confusion wrapper] + [target task replacement]`

**주요 결합 family**  
M01, B02, A01, A02, 일부 M04.

**권장 mutation**
paraphrase, delimiter break, fake role header, quoted instruction, summary/translation wrapper, style shift.

**권장 평가 레이어 / split**
- Layer 1, Layer 2, Layer 3
- public_train + benign_hard_negative

**근거·출처**
- 직접 근거: `S1`, `S2`
- 공개 근거(`S1` source manifest 계승): PromptInject, HouYi, direct seed taxonomy 계열
- 내부 보강: `I4`, `A1`, `A2`

---

### EN-02. 문서·웹·RAG 간접 인젝션 (indirect_document_rag)

**정의**  
외부 문서, 웹페이지, RAG chunk, 표, 코드블록, HTML/Markdown/JSON 안의 숨은 지시가 모델 동작을 바꾸는 유형.

**대표 attack surface**  
retrieved document, web content, tables, code comments, markdown/html/json/yaml, long-context chunk.

**권장 EN slot template**
- `[retrieved document wrapper] + [hidden steering sentence] + [benign user task]`
- `[table/cell content] + [embedded instruction-like phrase]`
- `[JSON/YAML/Markdown wrapper] + [comment or metadata directive]`

**주요 결합 family**  
M02, B01, M03, A04.

**권장 mutation**
footer injection, table-cell insertion, wrapper confusion, HTML/Markdown boundary confusion, long-context dilution, unicode/format noise.

**권장 평가 레이어 / split**
- Layer 2, Layer 3, Layer 4
- public_train + external_heldout

**근거·출처**
- 직접 근거: `S1`, `S2`
- 공개 근거(`S1` source manifest 계승): BIPIA, indirect prompt injection, web/RAG benchmark 계열
- 내부 보강: `I4`, `I5`, `A2`

### EN-03. 이메일 에이전트 인젝션 (email_agent)

**정의**  
메일 assistant가 메일 본문, 회신/전달 스레드, quoted reply 안의 숨은 지시를 신뢰해 잘못된 답변이나 행동을 하게 만드는 유형.

**대표 attack surface**  
mail body, thread history, footer, forwarding note, reminder line, assistant summary.

**권장 EN slot template**
- `[benign business email] + [quoted thread] + [hidden action request]`
- `[mail footer/note] + [instruction-looking text] + [normal summary task]`
- `[urgent business framing] + [seemingly routine approval/check request]`

**주요 결합 family**  
M02, A04, M04, 일부 A01.

**권장 mutation**
quoted reply injection, business register shift, urgency framing, footer disguise, polite escalation.

**권장 평가 레이어 / split**
- Layer 3, Layer 4
- external_heldout + private_adaptive

**근거·출처**
- 직접 근거: `S1`, `S2`
- 공개 근거(`S1` source manifest 계승): LLMail-Inject, indirect application prompt injection 계열
- 내부 보강: `I4`

### EN-05. 도구·에이전트 오남용 (tool_agent_misuse)

**정의**  
도구 출력, function call 인자, browser 결과, agent memory를 통해 텍스트 오염이 실제 행위 오용으로 이어지는 유형.

**대표 attack surface**  
tool output, function arguments, execution log, browser results, agent memory, code interpreter context.

**권장 EN slot template**
- `[tool output] treated as [higher-priority note]`
- `[function argument field] + [steering text]`
- `[execution log / artifact] + [behavior-changing sentence]`

**주요 결합 family**  
M02, B03, A02, 일부 B01.

**권장 mutation**
tool-result injection, argument poisoning, multi-step memory trigger, hidden instruction in retrieved tool output, hierarchy confusion.

**권장 평가 레이어 / split**

## 9. seed 설계 및 corpus 생성 규칙

### 9.1 최소 seed 구성 단위

각 seed는 최소한 아래 8개 필드를 가진다.

1. **attack_surface**
2. **goal**
3. **carrier_context**
4. **language_scope**
5. **style**
6. **mutation_family**
7. **oracle**
8. **recommended_layer**

### 9.2 추천 seed 생성식

```text
총 seed 수
= type 수
× base seed 수
× carrier 수
× 난이도 수
× language 수
× hard_negative 비율
```

### 9.3 현실적인 시작점

- type: EN 7 + KO 5
- base seed: 유형당 5개
- carrier: plain / document / RAG / tool / email / repo 중 유형별 3~4개
- difficulty: clean / paraphrase / obfuscation / structured wrapper
- language: en / ko / mixed

대략 다음과 같은 규모가 현실적이다.

```text
(12 type) × (5 base) × (4 carrier) × (4 difficulty) × (2 language)
= 약 1,920개 기본 seed
```

여기에 KO-native mutation pack(K01~K05)을 곱하면, 실전형 corpus가 된다.

## 12. 품질 게이트

좋은 퍼징 코퍼스는 수량보다 **구조**가 중요하다. 최소한 아래 조건을 지키는 것이 좋다.

1. **각 공격 seed마다 대응되는 benign/hard negative** 를 최소 1개 이상 둔다.
2. **EN-KO paired set** 은 same-goal same-position을 유지한다.
3. **KO-only set** 은 번역형이 아니라 한국어 고유 mutation을 반드시 포함한다.
4. **generator output** 은 public benchmark와 별도 split에 저장한다.
5. **tool/action 성공** 과 **text-only 성공** 을 분리 기록한다.
6. **wrapper/roleplay/urgency** 는 서로 다른 family로 기록한다.
7. **dedup** 은 문자열 기준이 아니라 의미 기준으로 한다.
8. **oracle** 이 불명확한 seed는 버린다.

| EN-06 | generator는 benchmark가 아니라 private adaptive set 생성기로 운용한다. |
| EN-07 | over-defense 측정이 빠지면 utility 손실을 놓친다. |
| KO-01 | paired set은 한국어 전체 성능이 아니라 “언어 차이”를 보기 위한 기준선이다. |
| KO-02 | 자모분리·초성·한글숫자는 반드시 최우선으로 넣는다. |
| KO-03 | 한국 서비스 carrier가 없으면 실제 운영형 취약점이 잘 드러나지 않는다. |
| KO-04 | 한국어 hard negative가 부족하면 false positive가 과소평가된다. |
| KO-05 | detector/gateway/agent를 한꺼번에 비교할 수 있도록 split discipline을 엄격히 유지한다. |

## 14. 최종 권고안

1. **영어는 7유형**, **한국어는 5유형**으로 고정하는 것이 가장 실용적이다.
2. **2차 자료의 family 카탈로그** 는 seed 확장 모듈로 유지하고, **1차자료의 EN/KO taxonomy** 는 최상위 구조로 사용한다.
3. 한국어는 반드시 **KO-native mutation layer** 를 별도 운영해야 한다.
4. **Layer 1~4** 프레임은 그대로 재사용하되, 실제 유출은 Layer 3~4에서 발생하므로 OUTPUT/RAG/멀티턴을 강조한다.
5. PromptFuzz/GPTFuzz/AgentVigil/PISmith/AutoInject/ChainFuzzer는 **generator-only** 로 관리한다.
6. PINT는 **held-out**, PrismData는 **bootstrap only**, prodnull는 **gated/raw redistribution 주의**, agent benchmark는 **external held-out**로 관리하는 편이 안정적이다.
