# 프롬프트 인젝션 퍼저 아키텍처 (수정: 2026-04-10 16:35:13 KST)

이 문서는 현재 저장소가 어떤 구조로 동작하는지 설명하는 아키텍처 문서다. 계획안이 아니라 현재 구현 기준 설명이며, source of truth는 코드와 현재 카탈로그/설정 파일이다.

## 1. 현재 아키텍처 한눈에 보기

```text
Authoring Catalog
  TemplateRecord JSONL
  CaseRecord JSONL
  generator configs + mutation recipes
  reference YAML catalogs
        |
        +--> generate-cases (optional)
        |     mvp: generated_cases.jsonl
        |     bulk: generated_bulk/{manifest,summary,pass_reports,indexes,shards}
        |
        v
build
  split 재배정
  dedup
  pair/linkage validation
  coverage gate
        |
        v
Build Package
  templates.jsonl
  cases.jsonl
  dedup_drops.jsonl
  manifest.json
        |
        v
run
  L1 / L2 / L3 runner
  target config + dispatch/map
        |
        v
RunRecord JSONL + transcripts
        |
        v
score
  detection/outcome/gateway metrics
  analysis buckets
        |
        v
Scorecard JSON
        |
        v
report
  Markdown report
  CSV raw results
```

핵심 흐름은 `Template -> Case -> Build Package -> Run -> Scorecard -> Report`다. 현재 실제 실행 단위는 `CaseRecord`와 `RunRecord`이고, `TemplateRecord`는 authoring 기준과 패키지 provenance를 유지하는 역할이 더 크다.

## 2. Authoring Catalog와 Build Package의 차이

### Authoring Catalog

- 편집 대상은 기본적으로 [`catalogs/sample_templates.jsonl`](catalogs/sample_templates.jsonl)과 [`catalogs/sample_cases.jsonl`](catalogs/sample_cases.jsonl)이다.
- generator를 사용하는 경우 [`configs/generator_mvp.yaml`](configs/generator_mvp.yaml), [`configs/generator_bulk.yaml`](configs/generator_bulk.yaml), [`catalogs/mutation_recipes.yaml`](catalogs/mutation_recipes.yaml)도 runtime 입력이 된다.
- 여기에 `attack_families.yaml`, `mutation_recipes.yaml`, `qa_policy.yaml`, `source_truth_manifest.yaml` 같은 보조 카탈로그가 붙는다.
- 현재 런타임에서 직접 읽는 reference YAML은 [`catalogs/analysis_taxonomy.yaml`](catalogs/analysis_taxonomy.yaml)과 [`catalogs/coverage_matrix.yaml`](catalogs/coverage_matrix.yaml)뿐이다.
- `attack_families.yaml`, `qa_policy.yaml`, `source_truth_manifest.yaml`, [`configs/run_matrix.yaml`](configs/run_matrix.yaml)은 현재 코드 경로에서 직접 로드되지 않는다. 현재 기준으로는 authoring/reference 성격이다.

### Generated Catalog

- [`catalogs/generated_cases.jsonl`](catalogs/generated_cases.jsonl)은 build가 함께 읽을 수 있는 파생 `CaseRecord` export다.
- `catalogs/generated_bulk/`는 bulk generator의 운영 디렉터리다. `manifest.json`, `summary.json`, `pass_reports/`, `indexes/`, `shards/`를 담는다.
- 이 두 경로는 curated source of truth가 아니라 generator가 다시 만들 수 있는 derived artifact다.

### Build Package

- build 산출물은 `templates.jsonl`, `cases.jsonl`, `dedup_drops.jsonl`, `manifest.json` 네 파일이다.
- build는 authoring catalog를 그대로 복사하지 않는다.
- `CaseRecord.split`은 build 시 deterministic rule로 다시 배정된다.
- dedup가 적용되고, validation과 coverage gate를 통과한 결과만 패키지에 남는다.
- `manifest.json`에는 패키지 메타데이터, source path, count, seed가 들어간다.

즉 authoring catalog는 편집용 원천 데이터이고, build package는 실행 가능한 스냅샷이다.

## 3. L1 / L2 / L3 역할 분리와 현재 구현 범위

현재 CLI가 실제로 실행하는 레이어는 `L1`, `L2`, `L3`뿐이다.

| 레이어 | 현재 역할 | 현재 연결 대상 |
|---|---|---|
| `L1` | text-only guardrail 검사 | [`configs/targets/text_http.yaml`](configs/targets/text_http.yaml) 또는 local fallback |
| `L2` | gateway/guardrail 응답 포함 검사 | [`configs/targets/gateway_http.yaml`](configs/targets/gateway_http.yaml) 또는 local fallback |
| `L3` | outcome scenario 실행 | [`configs/targets/scenario_local.yaml`](configs/targets/scenario_local.yaml) |

현재 CLI 동작은 다음처럼 분리된다.

- `L3`는 `template_kind == "outcome_scenario"`만 실행한다.
- `L1`, `L2`는 `template_kind != "outcome_scenario"`만 실행한다.
- `execution_layer=L4_e2e_rag`는 현재 `CaseRecord` 분석 라벨일 뿐, 실행 가능한 CLI 레이어가 아니다.

## 4. 왜 split 재배정, dedup, validation, coverage gate가 build에 묶여 있는가

현재 구조에서 build는 단순 포맷 변환이 아니라 실행 전 품질 게이트다.

- split 재배정: `semantic_equivalence_group`를 우선 쓰고, 비어 있으면 `template_id`를 fallback 키로 써서 같은 split에 묶고 contamination을 예방한다.
- dedup: 동일 payload와 near-dup를 실행 전에 제거해 score 왜곡을 줄인다.
- validation: pair drift, KR-EN link, benign sibling, source-role/stage linkage 오류를 실행 전에 차단한다.
- coverage gate: shipped package가 최소 coverage 요구를 만족하는지 빌드 시점에 확인한다.

이 네 단계가 build에 묶여 있는 이유는 현재 run/score가 package를 “이미 품질 보증된 실행 단위”로 가정하기 때문이다. `validate_package()`가 재검증을 제공하긴 하지만, 이는 운영용 점검 경로이며 build 자체가 첫 번째 gate다.

## 5. Target Config와 Runner 연결 방식

타깃 연결 경로는 다음과 같다.

1. CLI `run`이 target YAML을 읽어 `TargetConfig`로 로드한다.
2. runner가 case 메타데이터와 실행 옵션을 합쳐 변수 집합을 만든다.
3. [`src/pi_fuzzer/dispatch.py`](src/pi_fuzzer/dispatch.py)가 `body_template`에 변수를 치환해 요청 payload를 만든다.
4. HTTP 타깃이면 `dispatch_http()`가 요청을 보내고, `response_field_map`으로 응답을 필요한 필드에 매핑한다.
5. local 타깃이면 runner 내부 기본 시뮬레이션이 동작한다.

현재 shipped target의 의미는 분명하다.

- `text_http.yaml`: text-only detector endpoint
- `gateway_http.yaml`: pre/post detection, enforcement, telemetry를 매핑하는 gateway endpoint
- `scenario_local.yaml`: 외부 HTTP 없이 재현 가능한 로컬 시나리오 baseline

주의할 점도 있다.

- `request_field_map`은 현재 코드에서 사용되지 않는다. 실제 요청 구성은 `body_template` 치환으로만 이뤄진다.
- `supports_pre_post`, `supports_mask`, `supports_tool_log`, `supports_ttft`도 현재 러너 분기에는 직접 쓰이지 않는다. 현재는 descriptive metadata에 가깝다.

## 6. 분석 taxonomy, scoring, reporting이 개입하는 지점

### Taxonomy

- [`catalogs/analysis_taxonomy.yaml`](catalogs/analysis_taxonomy.yaml)을 [`src/pi_fuzzer/normalize.py`](src/pi_fuzzer/normalize.py)가 읽는다.
- source stage/role, interpretation, detector, failure, policy execution, tool transition, vendor claim gap 같은 값을 canonical space로 정규화한다.
- 빈 값은 그대로 두고, 비어 있지 않은 미등록 값은 대부분 `other`로 정규화한다.

### Runner

- [`src/pi_fuzzer/runners.py`](src/pi_fuzzer/runners.py)는 case 메타데이터를 그대로 복사하지 않는다.
- `source_role`, `expected_interpretation`, `tool_transition_type`, `policy_executed`, `failure_cause_code`, `decision_trace` 등을 기본 규칙으로 backfill한다.
- 따라서 sample case 일부가 빈 값을 갖더라도 run 단계에서 분석 축이 보강된다.

### Scoring

- [`src/pi_fuzzer/scoring.py`](src/pi_fuzzer/scoring.py)는 detection metric, outcome metric, gateway loss를 계산한다.
- 동시에 `by_source_role`, `by_expected_interpretation`, `by_detector_family`, `by_policy_request_vs_execution`, `by_tool_transition`, `by_vendor_claim_gap` 등 분석 bucket을 만든다.

### Reporting

- [`src/pi_fuzzer/reporting.py`](src/pi_fuzzer/reporting.py)는 scorecard JSON을 Markdown 표와 CSV로 내보낸다.
- Markdown은 사람용 집계표이고, CSV는 raw `RunRecord` 덤프다.

## 7. replay / tool-transition / KR-EN / benign sibling 축이 붙는 위치

이 축들은 현재 전부 `CaseRecord`에서 시작해 `RunRecord`와 `Scorecard`까지 이어진다.

### replay / tool-transition

- `source_stage`, `source_role`, `entry_point`, `tool_transition_type`, `replay_window`, `delayed_injection_turn`, `structured_payload_type`가 case에 붙는다.
- validation은 replay/tool_output/retrieval 계열 role/interp 규칙을 검사한다.
- dedup는 structural fingerprint에 이 축을 포함한다.
- runner는 `replay_turn_index`, `delayed_trigger_fired`를 runtime으로 채운다.
- scoring은 `by_tool_transition`, `by_guard_stage_alignment`, `by_contrast_group_outcome`에서 이 축을 읽는다.

### KR-EN pair

- 기본 pair는 `paired_case_id`로 연결된다.
- 명시적 bilingual pair group이 필요할 때는 `kr_en_pair_id`를 추가로 쓴다.
- validation은 `kr_en_pair_id`가 있으면 ko/en이 모두 존재하고 `paired_case_id`가 그 그룹 안에 닫혀 있는지 검사한다.
- scoring은 detection/outcome metric의 `kr_en_gap`와 contrast outcome의 `ko_en_gap`에서 이 축을 읽는다.

### benign sibling / contrast

- `benign_sibling_id`, `contrast_group_id`, `paired_case_role`가 case에 붙는다.
- validation은 benign sibling의 반대 label 여부와 contrast group 내 benign control 존재 여부를 검사한다.
- scoring은 `by_contrast_group_outcome`에서 attack/benign을 함께 집계한다.

## 8. 현재 구조의 한계와 P2 경계

현재 구조는 P0/P1 범위까지 구현돼 있다. 아래는 아직 현재 구현 완료처럼 읽으면 안 되는 항목이다.

- `L4_e2e_rag`는 실행 레이어가 아니라 현재도 분석 라벨이다.
- `run_matrix.yaml`은 현재 CLI에서 직접 읽지 않는다.
- placeholder-only는 shipped sample과 mutation recipe의 현재 운영 원칙이지, build/validate가 강제하는 기계 규칙은 아니다.
- `validate_package()`는 dedup를 다시 계산하지 않는다.
- `score` CLI는 `--config`를 주면 coverage 요약을 다시 계산해 scorecard에 넣고, 없으면 coverage를 미평가 상태로 기록한다.
- vendor claim gap은 heuristic bucket이지, 외부 vendor statement의 정식 검증기가 아니다.
- ARR/trajectory 고도 분석, memory laundering 심화, coverage policy 추가 강화는 P2 backlog다.
