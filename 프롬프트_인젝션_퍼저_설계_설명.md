# 프롬프트 인젝션 퍼저 설계 설명 (수정: 2026-04-10 16:35:13 KST)

이 문서는 현재 구조가 왜 이렇게 나뉘어 있는지 설명하는 rationale 문서다. 구현된 내용과 아직 구현되지 않은 P2 항목을 분리해서 적는다.

## 1. `Template / Case / Run / Scorecard`를 왜 분리했는가

### `TemplateRecord`

- 템플릿은 공격군 또는 benign control의 “형식적 골격”을 표현한다.
- 현재 shipped sample은 placeholder-only를 유지하므로, 템플릿은 concrete payload가 아니라 slot 구조와 분류 기준을 담는 데 적합하다.
- 현재 runtime은 주로 case를 실행하지만, template는 package provenance와 `template_kind` 기반 레이어 분기에 계속 필요하다.

### `CaseRecord`

- 케이스는 실제 실행 단위다.
- language, pair 관계, mutation, replay metadata, policy expectation, contrast linkage 같은 실험 축이 여기에 모인다.
- build, validation, dedup, scoring이 모두 case 단위로 작동하므로, template보다 세밀한 비교 envelope가 필요하다.

### `RunRecord`

- run은 “특정 case가 특정 target/config/enforcement mode에서 실제로 어떻게 실행됐는가”를 기록한다.
- case와 분리해 두면 같은 case를 여러 target, 여러 enforcement mode, 여러 guardrail toggle에서 반복 실행해도 원본 authoring metadata는 보존된다.
- 정책 실행 결과, detector 결과, response disposition, latency, transcript hash 같은 runtime-only 정보는 run 단계에서만 생긴다.

### `Scorecard`

- scorecard는 run을 집계한 결과물이다.
- detection metric, outcome metric, gateway loss, bucketized analysis를 한 파일에 담아 downstream report가 같은 입력을 공유하게 한다.
- score/report를 분리한 이유는 JSON scorecard를 재가공 가능한 canonical artifact로 두기 위해서다.

## 2. 왜 `source_stage / source_role / expected_interpretation` 축이 필요한가

`source_stage`만으로는 현재 구조가 표현하려는 신뢰 경계를 설명하기 부족하다.

- `source_stage`는 입력이 시스템 어디에서 들어왔는지 말해 준다.
- `source_role`은 그 입력이 어떤 역할로 소비돼야 하는지 말해 준다.
- `expected_interpretation`은 모델이나 도구가 그 입력을 `instruction`으로 읽어야 하는지, `data`로 읽어야 하는지, `analysis_target`이나 `quoted_text`로 읽어야 하는지 말해 준다.

이 셋을 분리한 이유는 다음과 같다.

- 같은 `retrieval`이어도 `retrieved_doc`와 `assistant_quote`는 다른 해석 경계를 가진다.
- 같은 `tool_input`이어도 `tool_output`을 data로 읽는 경우와 user form field를 workflow state로 읽는 경우는 실패 원인이 다르다.
- scoring에서 어떤 guard stage가 실패했는지 설명하려면 stage만이 아니라 “무엇으로 읽혔는지”가 필요하다.

현재 validation이 retrieval/tool_output/replay 계열에서 role/interp를 요구하는 것도 같은 이유다.

## 2.1 왜 generator를 build와 분리했는가

- generator는 candidate `CaseRecord`를 만드는 단계고, build는 최종 split/dedup/coverage truth를 만드는 단계다.
- 둘을 분리해 두면 `generated_cases.jsonl`을 opt-in source로 붙이면서도 기존 curated build 흐름을 깨지 않는다.
- `mode=mvp`는 flat JSONL export만 만들고, `mode=bulk`는 shard/index/pass report를 추가하지만 build 입력은 계속 flat export 하나로 유지한다.
- 이 구조 덕분에 generator preflight가 build semantics를 미리 흉내 내더라도 최종 판정 권한은 build에 남는다.

## 3. 왜 KR-EN pair와 benign sibling/contrast가 필요한가

### KR-EN pair

- 현재 저장소는 `kr_en_gap`를 detection/outcome metric에 직접 반영한다.
- 따라서 단일 언어 공격 성공률만으로는 충분하지 않고, 의미는 같은데 언어만 다른 케이스가 필요하다.
- `paired_case_id`는 기본 bilingual pairing을 제공하고, `kr_en_pair_id`는 replay/structured payload처럼 group-level linkage를 더 명확히 잡을 때 쓴다.

### benign sibling / contrast

- 공격 케이스만 있으면 false positive와 overblocking을 같은 맥락에서 읽기 어렵다.
- `benign_sibling_id`와 `contrast_group_id`를 두면 공격과 benign control을 같은 group 안에서 함께 읽을 수 있다.
- scoring의 `by_contrast_group_outcome`은 이 구조가 있어야 attack success와 benign overblock을 같이 보여 줄 수 있다.

즉 KR-EN pair는 언어 민감도 비교용이고, benign sibling/contrast는 공격 성공과 안전한 입력 오탐을 같은 실험 축에서 비교하기 위한 장치다.

## 4. 왜 replay / tool-transition 축이 별도 메타데이터인가

현재 구조는 “무슨 payload였는가”뿐 아니라 “그 payload가 어떤 경로로 도구/메모리/사용자 인터페이스로 이동했는가”를 분리해서 본다.

- `tool_transition_type`이 없는 상태에서는 tool output poisoning, approval form injection, replay delayed trigger를 모두 비슷한 payload로 오해하기 쉽다.
- `replay_window`, `delayed_injection_turn`이 없으면 delayed trigger는 단순 multi-turn attack과 구분되지 않는다.
- dedup에서도 같은 payload를 다른 transition envelope로 보존해야 하므로 이 축이 fingerprint에 들어간다.
- scoring에서 `by_tool_transition`과 `by_guard_stage_alignment`를 읽으려면 이 축이 case/run에 남아 있어야 한다.

현재 replay/tool-transition은 부가 설명용 메타데이터가 아니라 dedup, validation, scoring을 모두 바꾸는 1급 분석 축이다.

## 5. 왜 dedup이 `exact hash + structural fingerprint + hybrid similarity` 구조인가

단일 dedup 방식으로는 현재 실험 목적을 만족시키기 어렵다.

### exact hash

- 완전히 같은 payload 중복을 빠르게 제거한다.

### structural fingerprint

- payload 텍스트가 같아도 비교 envelope가 다르면 보존해야 한다.
- 현재 fingerprint는 `source_role`, `expected_interpretation`, `kr_en_pair_id`, `benign_sibling_id`, `contrast_group_id`, `tool_transition_type`, `replay_window`, `delayed_injection_turn`, `structured_payload_type`, `threshold_profile`, `normalization_variant`, `primary_mutation`, `secondary_mutations`, `mutation_family`를 포함한다.
- 이 구조 덕분에 “같은 문구지만 다른 trust boundary에 들어가는 경우”를 살아 있게 둘 수 있다.

### hybrid similarity

- release 용도에서는 text가 조금씩만 바뀐 near-dup도 제거해야 한다.
- 그래서 hybrid mode에서는 token-count cosine similarity를 추가로 본다.
- 현재 구현은 release build에서 `hybrid_mandatory`를 강제하고, 그 범위도 `heldout_static`, `adaptive` split으로 제한한다.

이 설계는 “실험 envelope는 살리고, 반복 payload는 줄인다”는 목표에 맞춰져 있다.

## 6. 왜 coverage gate가 profile 기반인가

현재 coverage gate는 단일 전역 규칙이 아니라 profile 집합으로 설계돼 있다.

- 기존 `required_dims + min_per_cell`만으로는 P0/P1 확장 축을 설명하기 어렵다.
- 어떤 빌드는 기본 커버리지만 보면 되고, 어떤 빌드는 replay/tool transition 조합까지 강제해야 한다.
- profile 구조로 두면 `required_dims`, `required_values`, `required_combinations`, `filters`, `enforce_cartesian`을 조합해 additive하게 강화할 수 있다.
- build config에서 어떤 profile을 쓸지 정하고, matrix YAML에서 이름으로 재사용할 수 있어 backward compatible 확장이 쉽다.

현재 shipped config가 `release_default`, `p0_stage_role`, `p1_replay_tool_transition`을 함께 읽는 이유도 이 구조 덕분이다.

## 7. 왜 `policy_requested / policy_executed / raw_policy_action`을 따로 두는가

현재 구조는 정책 계층을 세 수준으로 나눈다.

- `policy_requested`: case나 운영 설정이 기대한 정책 모드
- `policy_executed`: raw runtime 결과를 canonical policy execution으로 정규화한 값
- `raw_policy_action`: 공급자 또는 gateway가 실제로 내놓은 원본 action

이 셋을 분리한 이유는 다음과 같다.

- `alert`, `deny`, `sanitize` 같은 vendor-specific raw action을 그대로 보존해야 한다.
- 동시에 score bucket에서는 `annotate`, `block`, `mask`, `allow`로 canonical comparison이 가능해야 한다.
- requested와 executed를 분리해야 mismatch를 바로 집계할 수 있다.

즉 raw는 forensic 용도, executed는 canonical scoring 용도, requested는 기대 정책 용도다.

## 8. 왜 scoring이 단순 ASR만 보지 않는가

단순 ASR만 보면 현재 구조의 중요한 실패를 놓친다.

- `by_config_sensitivity`: 같은 case가 config별로 크게 달라지는지 본다.
- `by_vendor_claim_gap`: 공급자가 지원한다고 적은 축과 실제 outcome 사이의 간극을 본다.
- `by_policy_request_vs_execution`: 요청 정책과 실행 정책이 일치했는지 본다.
- `by_contrast_group_outcome`: 공격 success와 benign overblock을 같은 group에서 함께 본다.
- `by_guard_stage_alignment`: 원래 어느 stage에서 막혀야 했는지와 실제 실패 stage를 비교한다.

즉 현재 scoring은 “공격이 성공했는가”를 넘어서 “왜 그런 결과가 나왔는가”를 읽기 위한 구조다.

## 9. 현재 구현된 범위

현재 코드에서 확인되는 범위는 여기까지다.

- `TemplateRecord`, `CaseRecord`, `RunRecord`, `Scorecard` 스키마 확장
- taxonomy canonicalization
- role/interp linkage validation
- KR-EN pair / benign sibling / contrast validation
- exact + structural + hybrid dedup
- profile-based coverage gate
- replay/tool-transition/runtime policy normalization
- config sensitivity / vendor claim gap / contrast outcome / guard stage alignment scoring

## 10. 아직 P2로만 남아 있는 항목

아래는 현재 구현 완료처럼 서술하면 안 되는 항목이다.

- ARR / trajectory 고도 분석
- memory laundering 심화 축 확장
- vendor claim-vs-measure 비교 정교화
- coverage policy를 더 강하게 올릴 추가 profile 운영 결정
- 실행 가능한 `L4` 경로
