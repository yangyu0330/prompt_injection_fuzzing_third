## 2. 패키지 구조와 현재 데이터 모델

현재 build 산출물은 아래 네 파일이다.

- `templates.jsonl`
- `cases.jsonl`
- `dedup_drops.jsonl`
- `manifest.json`

generator를 쓸 때는 추가 파생 산출물이 생긴다.

- `catalogs/generated_cases.jsonl`: build가 읽을 수 있는 generated case export
- `catalogs/generated_bulk/manifest.json`: bulk run fingerprint와 shard/index 메타데이터
- `catalogs/generated_bulk/summary.json`: 최종 status와 survivor/deficit 요약
- `catalogs/generated_bulk/pass_reports/pass-XXXX.json`: pass별 family/coverage 요약
- `catalogs/generated_bulk/indexes/*.jsonl`: bulk dedup 보조 색인
- `catalogs/generated_bulk/shards/family=.../part-XXXX.jsonl`: family별 committed shard

중요:

- `generated_cases.jsonl`과 `generated_bulk/`는 source of truth가 아니라 generator가 재구성 가능한 파생 산출물이다.
- 최종 split/dedup/coverage 판정은 여전히 build 결과를 기준으로 본다.

샘플 카탈로그는 `Template -> Case -> RunRecord -> Scorecard` 순으로 연결된다.

### 2.1 TemplateRecord

`TemplateRecord`는 공격/benign 유형과 허용 가능한 realization 범위를 표현한다. 현재 핵심 필드는 아래와 같다.

- 기본 구조: `template_id`, `template_kind`, `attack_or_benign`, `attack_family`, `attack_subfamily`, `directness`, `source_stage`, `turn_mode`, `supported_entry_points`, `supported_carriers`
- 보호 목표/오라클: `protected_asset`, `forbidden_action`, `expected_safe_behavior`, `tool_scope`, `oracle_type`, `severity`
- 동등성/확장 축: `semantic_equivalence_group`, `allowed_mutation_families`, `supported_source_roles`, `supported_expected_interpretations`
- P1 축: `tool_transition_type`, `replay_enabled`, `delayed_trigger`, `vendor_declared_support`

### 2.2 CaseRecord

`CaseRecord`는 build와 scoring의 핵심 분석 단위다. 현재 스키마는 초기에 비해 분석용 필드가 크게 확장되어 있다.

- 비교/링크 축: `paired_case_id`, `kr_en_pair_id`, `benign_sibling_id`, `contrast_group_id`, `paired_case_role`
- 입력 해석 축: `source_stage`, `source_role`, `expected_interpretation`, `entry_point`, `carrier_context`
- 한국어/형태 축: `mutation_family`, `primary_mutation`, `secondary_mutations`, `register_level`, `script_mix_profile`, `normalization_required`
- 분석/원인 축: `analysis_axis`, `context_dependency_type`, `expected_guard_stage`, `execution_layer`, `primary_target_entity`, `expected_sensitive_entities`
- 정책/탐지 축: `policy_requested`, `detector_family`, `failure_cause_code`, `reason_codes`, `matched_rule_ids`, `decision_trace`, `config_fingerprint`
- 사용자 영향 축: `final_user_visible`, `unsafe_tool_call`, `partially_allowed`, `partial_defense`
- replay/tool-transition 축: `tool_transition_type`, `replay_window`, `delayed_injection_turn`, `structured_payload_type`, `threshold_profile`, `normalization_variant`, `vendor_declared_support`

중요한 점은 모든 sample row가 이 필드를 전부 직접 채우지는 않는다는 점이다. 현재 구현은 backward compatible 동작을 유지한다. 고위험 stage나 P1 샘플은 필드를 직접 채우고, legacy 성격의 row는 실행 시 러너가 기본값을 보강한다.

### 2.3 RunRecord

`RunRecord`는 실제 실행 결과를 담는다. 현재 러너가 채우는 핵심 축은 다음과 같다.

- 기본 실행: `layer`, `target_id`, `system_config_id`, `repeat_index`, `guardrail_toggle`, `enforcement_mode`
- detection/outcome: `detected_pre`, `detected_post`, `masked`, `blocked_effectively`, `effective_pass`, `secret_exposed`, `unauthorized_tool_used`, `task_hijacked`, `task_completed_safely`, `benign_overblocked`
- 입력 정규화: `observed_input_text`, `normalized_input_text`, `applied_normalizers`, `normalization_changed`, `normalization_diff_tags`, `language_route`
- 정책/탐지 결과: `policy_requested`, `policy_executed`, `raw_policy_action`, `detector_family`, `failure_cause_code`, `reason_codes`, `matched_rule_ids`, `decision_trace`, `config_fingerprint`
- 사용자 영향: `final_user_visible`, `unsafe_tool_call`, `partially_allowed`, `partial_defense`, `response_disposition`
- replay/tool-transition: `tool_transition_type`, `replay_turn_index`, `delayed_trigger_fired`, `chunk_join_required`, `chunk_join_succeeded`
- 분석 보조: `failure_stage`, `failure_reason_tags`, `tool_decision_source`, `threshold_profile`, `normalization_variant`, `vendor_declared_supported`

### 2.4 Scorecard

`Scorecard`는 headline metric과 분석 bucket을 함께 담는다.

- 기본 metric: `recall`, `precision`, `fpr`, `bor`, `kr_en_gap`, `asr`, `ser`, `uar`, `stcr`, `integration_loss`, `enforcement_loss`
- 기존 bucket: `by_layer`, `by_attack_family`, `by_mutation`, `by_entry_point`, `by_source_stage`, `by_turn_mode`, `by_guardrail_toggle`, `by_enforcement_mode`, `by_lang`
- P0/P1 bucket: `by_source_role`, `by_expected_interpretation`, `by_detector_family`, `by_failure_cause_code`, `by_policy_request_vs_execution`, `by_raw_policy_action`, `by_reason_code`, `by_tool_transition`, `by_config_sensitivity`, `by_vendor_claim_gap`
- 추가 분석 bucket: `by_final_user_visible`, `by_analysis_axis`, `by_primary_mutation`, `by_register_level`, `by_failure_stage`, `by_language_route`, `by_primary_target_entity`, `by_execution_layer`, `by_policy_mode`, `by_contrast_group_outcome`, `by_guard_stage_alignment`

## 3. 현재 build 흐름

현재 `build_package()` 흐름은 아래 순서로 고정되어 있다.

```text
load templates/cases
-> deterministic split assign
-> dedup
-> pair/split/analysis validation
-> coverage profile enforcement
-> package export
```

세부 동작은 다음과 같다.

- template/case source는 config에 지정된 JSONL을 읽는다. 지정이 없으면 `catalogs/sample_templates.jsonl`, `catalogs/sample_cases.jsonl`를 기본값으로 사용한다.
- split은 `semantic_equivalence_group`가 있으면 그 값을, 없으면 `template_id`를 기준으로 seed 기반 deterministic assignment를 한다.
- release build는 `dedup.mode=hybrid_mandatory`가 아니면 실패한다.
- release build dedup은 `heldout_static`, `adaptive` split에만 hybrid dedup을 적용하고, 그 외 split은 그대로 둔다.
- build 실패 조건은 validation 에러 또는 coverage violation이다.
- 성공 시 `templates.jsonl`, `cases.jsonl`, `dedup_drops.jsonl`, `manifest.json`을 출력한다.

## 4. validation과 dedup의 현재 규칙

### 4.1 pair/linkage validation

현재 validation은 다음 세 층으로 나뉜다.

- `validate_pair_invariants`
- `validate_split_contamination`
- `validate_analysis_linkage`

`validate_pair_invariants`는 KR/EN pair에서 아래 필드가 drift 되지 않아야 통과한다.

- `user_goal`
- `protected_asset`
- `forbidden_action`
- `tool_scope`
- `entry_point`
- `severity`

`validate_analysis_linkage`는 아래 세 규칙을 묶는다.

- `validate_kr_en_pair_links`: `kr_en_pair_id`가 있으면 ko/en이 모두 존재하고, `paired_case_id`가 같은 pair 안에서 닫혀 있어야 한다.
- `validate_benign_sibling_and_contrast`: `benign_sibling_id`는 반대 label을 가리켜야 하고, contrast group에서 benign control이 필요한데 빠져 있으면 실패한다.
- `validate_source_role_stage_coverage`: `source_stage=retrieval`, `tool_input`, `tool_output`, `replay`이거나 role/interp를 직접 쓰는 경우 `source_role`, `expected_interpretation`가 일관돼야 한다.

### 4.2 dedup

현재 dedup은 `dedup_cases()` 하나로 관리한다.

- exact hash: `trusted_instruction`, `user_goal`, `untrusted_content`, `expected_safe_behavior`를 normalize한 payload가 같고 comparison envelope도 같으면 drop한다.
- structural fingerprint: payload가 같지 않아도 envelope가 같으면 drop한다.
- hybrid mode일 때는 token-count cosine similarity가 threshold 이상인 near-dup도 drop한다.

중요한 구현 포인트는 structural fingerprint에 다음 축이 포함된다는 점이다.

- `source_role`
- `expected_interpretation`
- `kr_en_pair_id`
- `benign_sibling_id`
- `contrast_group_id`
- `tool_transition_type`
- `replay_window`
- `delayed_injection_turn`
- `structured_payload_type`
- `threshold_profile`
- `normalization_variant`
- `primary_mutation`
- `secondary_mutations`
- `mutation_family`

따라서 같은 payload라도 비교 envelope가 다르면 보존된다. 현재 테스트도 이 동작을 확인한다.

## 5. coverage enforcement

현재 coverage gate는 `coverage_gate.profiles`를 중심으로 동작한다. profile은 config에 inline으로 넣을 수도 있고, `catalogs/coverage_matrix.yaml`의 이름을 참조할 수도 있다.

### 5.1 profile 구성 요소

- `required_dims`: coverage cell을 구성하는 차원
- `min_per_cell`: 각 cell 최소 개수
- `filters`: 특정 subset에만 gate 적용
- `required_values`: 특정 축의 특정 값이 최소 몇 번 이상 있어야 하는지 보장
- `required_combinations`: 특정 조합이 반드시 존재해야 하는지 보장
- `enforce_cartesian`: `required_dims` 전체에 대해 cartesian cell을 강제할 때 사용

### 5.2 현재 기본 config에서 실제로 쓰는 profile

`configs/build_dev.yaml`, `configs/build_release_heldout.yaml`은 현재 세 개 profile을 사용한다.

| profile | 의미 |
|---|---|
| `release_default` | `language x source_stage x directness x attack_family` 축에서 현재 관측된 cell의 최소 개수를 검사하는 기본 커버리지 |
| `p0_stage_role` | `source_role`, `expected_interpretation`가 실제 샘플에 존재하는지와 핵심 조합이 있는지 확인 |
| `p1_replay_tool_transition` | multi-turn replay/tool transition 축이 실제 샘플에 존재하는지 확인 |

주의할 점이 있다. 현재 구현에서 `required_dims + min_per_cell`만으로는 비어 있는 전체 cartesian 조합을 자동으로 강제하지 않는다. 빠진 조합까지 보장하려면 `required_combinations` 또는 `enforce_cartesian`을 함께 써야 한다.

추가로 `coverage_matrix.yaml`에는 아직 기본 build에 켜지지 않은 profile도 있다.

- `l2_runs`
- `ko_native_axes`
- `ko_mutation_focus`

### 5.3 release와 dev의 범위 차이

- dev validate: 모든 split을 본다.
- release validate: `heldout_static`, `adaptive`만 coverage 대상으로 본다.

이 차이는 현재 테스트로 고정돼 있다.

## 6. 러너가 실제 분석 축을 채우는 방식

현재 `runners.py`는 case 메타데이터를 그대로 복사하는 데서 끝나지 않는다. HTTP 응답 mapping 또는 local baseline 결과와 결합해 아래 규칙으로 runtime 필드를 채운다.

- `source_role`: `entry_point` 기준 기본값을 계산한다. 예를 들어 `tool_output -> tool_output`, `memory_or_summary -> memory_note`, `quoted_reply -> assistant_quote`
- `expected_interpretation`: `tool_output`, `memory_or_summary`, `form_field`, `file_text`는 기본값이 `data`다.
- `tool_transition_type`: `source_stage=replay -> replay_to_tool`, `entry_point=tool_output -> tool_to_user`, `memory_or_summary -> memory_to_tool`
- `policy_executed`: `raw_policy_action`이 문자열, 객체, 배열 중 무엇이든 canonical policy execution으로 변환한다.
- `failure_cause_code`: 응답이 직접 주지 않으면 `policy_miss`, `execution_gap`, `normalization_miss`를 상태에서 유도한다.
- `decision_trace`: 응답이 직접 주지 않으면 detection/block/response disposition으로 기본 trace를 만든다.

따라서 현재 구현은 sample case의 일부 필드가 비어 있어도 score/report 수준의 분석 bucket을 유지한다.

## 7. scoring과 reporting의 현재 관계

현재 scoring은 run과 case를 같이 본다. case는 의도와 기대값을 제공하고, run은 실제 실행 결과를 제공한다.

- detection metric은 `detected_pre`, `detected_post`, `blocked_effectively`를 기준으로 계산한다.
- outcome metric은 `secret_exposed`, `unauthorized_tool_used`, `task_hijacked`, `task_completed_safely`를 기준으로 계산한다.
- gateway loss는 L1 recall과 L2 recall 차이, 그리고 detection 대비 실제 block 차이로 계산한다.
- `by_policy_request_vs_execution`은 requested/executed mismatch를 바로 집계한다.
- `by_raw_policy_action`은 문자열뿐 아니라 object/array action도 그대로 bucket으로 남긴다.
- `by_reason_code`는 `reason_codes`가 비어 있으면 pre/post detector reason code를 합쳐 대신 쓴다.
- `by_config_sensitivity`는 같은 case가 여러 `config_fingerprint`에서 얼마나 결과가 달라졌는지 본다.
- `by_vendor_claim_gap`는 case의 `vendor_declared_support` 또는 run의 `vendor_declared_supported`를 실제 공격 성공과 비교해 `claim_match`, `claim_gap`, `not_declared`로 분류한다.

reporting은 scorecard의 이 bucket을 Markdown 표와 CSV로 그대로 노출한다. Markdown 리포트에는 현재 아래 섹션이 있다.

- By Source Stage / Source Role / Expected Interpretation
- By Detector Family / Failure Cause Code
- By Policy Mode / Policy Request vs Execution / Raw Policy Action
- By Reason Code / Tool Transition / Final User Visible
- By Config Sensitivity / Vendor Claim Gap
- By Analysis Axis / Primary Mutation / Register Level / Failure Stage
- By Contrast Group Outcome / Guard Stage Alignment

## 8. shipped sample catalog의 현재 반영 상태

현재 기본 sample catalog는 placeholder-only sample이지만, 분석 축 시연용으로는 이미 P0/P1 범위를 반영한다.

- 템플릿 수: `24`
- 케이스 수: `36`
- template kind 분포: `diagnostic_attack 13`, `benign_hard_negative 2`, `outcome_scenario 9`
- source stage 분포: `input`, `retrieval`, `tool_input`, `tool_output`, `output`, `replay`
- 한국어 중심 mutation/sample이 많지만 EN pair도 일부 유지한다.

현재 상태를 해석할 때 중요한 점은 아래와 같다.

- `paired_case_id` 기반 KR/EN 연결은 광범위하게 쓰이고 있다.
- explicit `kr_en_pair_id`는 현재 P1 replay pair와 P1 structured payload pair에 적용되어 있다.
- benign sibling은 현재 replay delayed trigger contrast group에서 실제로 연결돼 있다.
- replay/tool-transition 축은 `CASE-KO/EN-P1-REPLAY-*`, `CASE-KO/EN-P1-STRUCT-*`에서 직접 채워져 있다.
- `catalogs/attack_families.yaml`와 `catalogs/analysis_taxonomy.yaml`는 sample보다 넓은 canonical space를 담고 있다. 즉 taxonomy coverage와 sample coverage는 동일 개념이 아니다.

## 9. 초기 설계 문서와 달라진 현재 기준

기존 설계 메모와 비교할 때 현재 구현에서 달라진 핵심은 아래다.

- 문서 중심 계획안이 아니라 `models.py`, `build.py`, `validation.py`, `runners.py`, `scoring.py`가 실제 기준이다.
- `source_stage=input|output` 수준이 아니라 `retrieval`, `tool_input`, `tool_output`, `replay`까지 first-class axis가 됐다.
- coverage는 단순 최소 cell 수가 아니라 named profile, required value, required combination 수준까지 올라왔다.
- dedup은 단순 exact dedup이 아니라 structural fingerprint와 hybrid near-dup 검사를 포함한다.
- replay/tool-transition과 policy execution mismatch가 score/report 단계까지 직접 이어진다.
- `execution_layer=L4_e2e_rag`는 현재 실행 layer가 아니라 분석 라벨이다. CLI는 여전히 `L1/L2/L3`만 실행한다.
