# Generator 평가비교묶음 설계

## 1. 문서 목적

이 문서는 generator의 `대량 생성 구현 계획`과 별도로,
나중 평가를 쉽게 하기 위한 `비교 묶음 규칙`을 정리한다.

핵심은 아래 두 가지를 분리하는 것이다.

- 생성 엔진이 실제로 관리하는 운영 단위
- 나중 평가와 비교를 위해 덧붙이는 해석 단위

이 문서는 두 번째를 다룬다.

## 2. 왜 별도 문서로 분리하는가

대량 생성 구현 계획 문서는 아래에 집중해야 한다.

- shard 저장
- export JSONL
- bundle 생성
- 중복 관리
- 재개 실행
- 부족분 재보충

반면 비교 묶음 규칙은 아래에 집중한다.

- 한국어와 영어를 어떻게 비교할지
- 정상 대비를 어떻게 붙일지
- 어떤 family에 이 비교 규칙이 자연스럽게 맞는지
- 어떤 family에는 강제하면 안 되는지

두 내용을 한 문서에 섞으면 구현 순서와 평가 해석 규칙이 뒤엉킨다.
그래서 이 문서를 별도로 둔다.

## 3. 기본 원칙

### 3.1 생성의 기본 뼈대는 지금 구조를 유지한다

현재 generator는 축 중심 조합으로 케이스를 만든다.

중요한 축은 아래와 같다.

- 입력 지점
- 전달 형식
- 역할
- 해석 기대
- 변형 계열
- 도구 전이
- 재생 구간

이 구조는 나중 평가를 위해서도 중요하다.
따라서 비교 묶음 규칙이 이 구조를 대체하면 안 된다.

### 3.2 비교 묶음은 생성 규칙이 아니라 평가 규칙이다

비교 묶음은 `generator의 기본 생성 단위`가 아니다.

비교 묶음은 아래 목적을 가진다.

- 영어 기본형과 한국어 대응형을 쉽게 비교
- 변형 전후를 쉽게 비교
- 정상 대비 여부를 쉽게 비교

즉 비교 묶음은 `결과 해석을 쉽게 만드는 틀`이다.

### 3.3 운영 단위와 평가 단위를 분리한다

- 운영 단위: bundle
- 평가 단위: 비교 묶음

bundle은 생성과 저장과 중복 관리에 필요하다.
비교 묶음은 보고서와 나중 평가에 필요하다.

둘을 같은 것으로 두지 않는다.

## 4. 권장 비교 묶음 틀

직접 입력형 또는 문장형 family에서는 아래 5종 틀이 유용하다.

1. 영어 기본형
2. 영어 변형형
3. 한국어 대응형
4. 한국어 변형형
5. 정상형

여기서 `한국어 대응형`은 반드시 직역 번역만 뜻하지 않는다.
서비스 문맥상 자연스러운 한국어 대응 표현도 포함한다.

## 5. 이 5종 틀을 언제 쓰는가

### 5.1 잘 맞는 경우

아래처럼 `문장 자체의 비교`가 중요한 family에서는 잘 맞는다.

- 직접 입력형
- 단일 문장 지시형
- 짧은 텍스트 우회형
- 번역 대응 비교가 의미 있는 family

이 경우에는 영어 기본형과 한국어 대응형을 붙여 비교하는 것이 평가 해석에 도움이 된다.

### 5.2 바로 강제하면 안 되는 경우

아래처럼 `언어 자체보다 입력 위치와 역할이 더 중요한 family`에는 일괄 강제하면 안 된다.

- 검색 문서형
- 도구 출력형
- 메모리 재생형
- 다단계 전이형
- 궤적 재생형

이 경우에는 먼저 아래가 더 중요하다.

- 입력 위치가 같은지
- 역할이 같은지
- 해석 기대가 같은지
- 전이 구조가 같은지

즉 이런 family는 5종 틀보다 기존 축 일치가 우선이다.

## 6. 비교 묶음을 만드는 기준

비교 묶음을 만들 때는 아래를 먼저 맞춘다.

- 같은 family
- 같은 template 계열
- 같은 입력 지점
- 같은 전달 형식
- 같은 역할
- 같은 해석 기대
- 같은 변형 계열 또는 대응 변형 계열

그 위에서 언어와 정상 대비를 붙인다.

즉 순서는 아래다.

1. 먼저 같은 문제인지 확인
2. 그 다음 영어와 한국어를 맞춘다
3. 그 다음 변형 전후를 맞춘다
4. 마지막으로 정상형을 붙인다

## 7. 정상형의 위치

정상형은 단순히 개수를 맞추기 위한 장식이 아니다.

정상형이 필요한 이유는 아래와 같다.

- 잘 막는 것과 과하게 막는 것을 구분
- 공격형과 정상형을 함께 비교
- family별 오탐 경향을 확인

따라서 비교 묶음에서 정상형은 가능하면 같이 붙이는 것이 좋다.
다만 자연스러운 정상형 template이 없는 family는 억지로 붙이지 않는다.

## 8. 적용 방식

비교 묶음은 한 번에 generator 핵심 규칙으로 넣지 않는다.
아래 순서로 붙이는 것이 안전하다.

### 8.1 1단계: 보고서용 표시만 추가

bulk mode가 안정화되면 먼저 report에서 비교 가능한 row들을 묶어 표시한다.

이 단계에서는:

- 실제 생성 규칙은 바꾸지 않는다
- 어떤 family가 비교 묶음에 잘 맞는지 확인한다
- 누락되는 묶음이 얼마나 많은지 본다

### 8.2 2단계: 일부 family에만 선택 적용

직접 입력형이나 문장형 family에 한해 비교 묶음 생성을 보조 규칙으로 추가한다.

이 단계에서는:

- 모든 family에 일괄 강제하지 않는다
- 적용 대상 family를 설정 파일로 제한한다
- 기존 bundle과 linkage 계약을 깨지 않는다

### 8.3 3단계: 평가 리포트에 정식 반영

비교 묶음이 충분히 안정화되면 아래를 리포트에 포함한다.

- 영어 기본형 대비 한국어 대응형 생존율
- 변형 전후 생존율
- 정상형 대비 오탐 비율
- family별 비교 묶음 완성도

## 9. 구현 계획 문서와의 관계

`generator_대량생성_구현_계획.md`는 아래를 먼저 끝내는 문서다.

- bulk mode 진입
- shard 저장
- export
- preflight
- 색인
- resume
- refill

이 문서는 그 다음에 붙일 평가 해석 규칙 문서다.

즉 순서는 아래가 맞다.

1. 대량 생성 모드 구현
2. survivor 2000 수준 안정화
3. 비교 묶음 표시
4. 일부 family에 선택 적용
5. 평가 리포트 정식 반영

## 10. 하지 않을 것

이 문서의 비교 묶음 규칙으로 아래를 바로 강제하지 않는다.

- 모든 family에 동일한 5종 틀 강제
- 직역 번역만 허용
- bundle과 비교 묶음을 같은 키로 취급
- 비교 묶음 완성 여부를 bulk 1차 완료 조건으로 사용

## 11. 한 줄 요약

비교 묶음은 generator의 기본 생성 규칙이 아니라,
bulk 구조가 안정화된 뒤 평가를 더 쉽게 하기 위해 덧붙이는 별도 규칙이다.
