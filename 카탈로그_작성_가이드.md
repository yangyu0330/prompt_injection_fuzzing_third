# 카탈로그 작성 가이드 (수정: 2026-04-10 16:35:13 KST)

이 문서는 새 template/case를 추가하는 작성자용 실무 가이드다. source of truth는 현재 코드와 샘플 카탈로그다. 구현되지 않은 미래 스키마를 가정하지 말고, 현재 필드만 사용한다.

## 1. 현재 작성 대상 파일

### 필수 입력

- [`catalogs/sample_templates.jsonl`](catalogs/sample_templates.jsonl)
- [`catalogs/sample_cases.jsonl`](catalogs/sample_cases.jsonl)

### generator 설정과 파생 산출물

- [`configs/generator_mvp.yaml`](configs/generator_mvp.yaml)
- [`configs/generator_bulk.yaml`](configs/generator_bulk.yaml)
- [`configs/build_generated_dev.yaml`](configs/build_generated_dev.yaml)
- [`catalogs/generated_cases.jsonl`](catalogs/generated_cases.jsonl): generator export, 수동 편집 금지
- `catalogs/generated_bulk/`: bulk manifest/shard/index/pass report working set, 수동 편집 금지

### 참고용 카탈로그

- [`catalogs/attack_families.yaml`](catalogs/attack_families.yaml): family 이름 정렬용 reference
- [`catalogs/mutation_recipes.yaml`](catalogs/mutation_recipes.yaml): `mutation_recipe_id`와 placeholder rule reference
- [`catalogs/analysis_taxonomy.yaml`](catalogs/analysis_taxonomy.yaml): canonical value space
- [`catalogs/coverage_matrix.yaml`](catalogs/coverage_matrix.yaml): build coverage profile reference
- [`catalogs/qa_policy.yaml`](catalogs/qa_policy.yaml): annotation/QA 메모
- [`catalogs/source_truth_manifest.yaml`](catalogs/source_truth_manifest.yaml): 문서 우선순위 메모

주의할 점:

- 현재 build/run/score가 직접 읽는 reference YAML은 `analysis_taxonomy.yaml`, `coverage_matrix.yaml`뿐이다.
- `generate-cases`는 여기에 더해 `mutation_recipes.yaml`과 generator config를 직접 읽는다.
- `attack_families.yaml`, `qa_policy.yaml`, `source_truth_manifest.yaml`은 현재 runtime validator가 자동 강제하지 않는다.
- 따라서 authoring 시 수동 정합성이 필요하다.

## 2. `sample_templates.jsonl`와 `sample_cases.jsonl`의 역할

### `sample_templates.jsonl`

- 공격군 또는 benign control의 “골격”을 정의한다.
- 현재는 generator의 seed catalog 역할도 한다.
- 분류, supported entry point/carrier, skeleton slot, 보호 자산, 오라클, semantic equivalence group을 잡는 데 쓴다.
- 현재 runtime에서 직접 소비되는 핵심 템플릿 필드는 `template_id`, `template_kind` 정도다.
- 나머지는 authoring/provenance/reference 성격이 강하다.

### `sample_cases.jsonl`

- 실제 build/run/score 단위다.
- language, pairing, mutation, source role, contrast, replay metadata, policy expectation이 모두 case에 붙는다.
- validation, dedup, coverage, scoring은 모두 case 중심이다.

실무 기준으로는 template보다 case 정합성이 더 중요하다.

### `generated_cases.jsonl`

- generator가 만든 파생 `CaseRecord` export다.
- build에서는 `sample_cases.jsonl`과 나란히 `case_sources`로 읽을 수 있다.
- source of truth가 아니라 derived artifact다. 직접 편집하지 않고 template/config/recipe를 수정한 뒤 다시 생성하는 편이 맞다.

## 3. placeholder-only 원칙

현재 shipped sample과 mutation recipe는 모두 placeholder-only다.

### 이렇게 작성하라

- `trusted_instruction`, `user_goal`, `untrusted_content`, `expected_safe_behavior`에는 `<...PLACEHOLDER>` 형태 slot만 넣는다.
- mutation recipe의 `rule`도 placeholder만 유지한다.
- 실제 공격 코퍼스나 재현 가능한 공격 문자열은 문서/샘플 JSONL에 복원하지 않는다.

### 이렇게 쓰지 말라

- 실제 공격 문장을 샘플 case에 넣지 말라.
- “예시니까 괜찮다”는 이유로 부분 payload를 넣지 말라.
- placeholder를 설명 없이 자연어 본문으로 바꿔 넣지 말라.

중요:

- 현재 build/validate는 placeholder-only를 기계적으로 강제하지 않는다.
- 이 원칙은 현재 저장소 운영 규칙이며, authoring review로 지켜야 한다.

## 4. Template 작성 규칙

### 필수 필드

다음 필드는 현재 `TemplateRecord`에서 required다.

- `template_id`
- `template_kind`
- `attack_or_benign`
- `attack_family`
- `attack_subfamily`
- `directness`
- `source_stage`
- `turn_mode`
- `supported_entry_points`
- `supported_carriers`
- `user_goal_skeleton`
- `trusted_instruction_skeleton`
- `untrusted_content_skeleton`
- `protected_asset`
- `forbidden_action`
- `expected_safe_behavior`
- `tool_scope`
- `oracle_type`
- `severity`
- `recommended_layer`
- `semantic_equivalence_group`
- `source_origin`
- `license_note`
- `redistribution_policy`

### 권장 필드

- `allowed_mutation_families`
- `supported_source_roles`
- `supported_expected_interpretations`
- `tool_transition_type`
- `replay_enabled`
- `delayed_trigger`
- `vendor_declared_support`
- `notes`

### 작성 원칙

- `template_kind`는 현재 구현상 `diagnostic_attack`, `benign_hard_negative`, `outcome_scenario`만 허용된다.
- `semantic_equivalence_group`는 케이스와 같이 재사용할 비교 단위를 미리 잡는다.
- `supported_entry_points`와 `source_stage`는 서로 모순되게 쓰지 말라.
- 현재 테스트는 non-input entry point를 가진 template가 `source_stage=input`로 남는 것을 허용하지 않는다.
- `supported_source_roles`, `supported_expected_interpretations`는 현재 validation에 직접 쓰이지 않지만, non-user source의 intent를 명확히 하는 authoring 힌트로 유지하는 편이 좋다.

## 5. Case 작성 규칙

### 필수 필드

현재 `CaseRecord`에서 required인 핵심 필드는 아래다.

- 식별/분류: `template_id`, `case_id`, `language`, `attack_or_benign`, `attack_family`, `attack_subfamily`, `directness`
- 입력 경로: `source_stage`, `turn_mode`, `entry_point`, `carrier_context`
- 본문: `user_goal`, `trusted_instruction`, `untrusted_content`, `expected_safe_behavior`
- 안전 기준: `tool_scope`, `severity`
- 운영 메타데이터: `split`, `source_origin`

### 권장 필드

현재 샘플과 scoring에서 사실상 자주 쓰는 권장 필드는 아래다.

- `semantic_equivalence_group`
- `mutation_family`
- `protected_asset`
- `forbidden_action`
- `oracle_type`
- `primary_mutation`
- `analysis_axis`
- `expected_guard_stage`
- `execution_layer`
- `primary_target_entity`
- `expected_sensitive_entities`
- `contrast_group_id`
- `paired_case_role`

### 작성 원칙

- `template_id`는 반드시 존재하는 template를 가리켜라.
- `attack_family`와 `attack_subfamily`는 template와 맞춰라.
- `directness`, `source_stage`, `turn_mode`, `entry_point`, `carrier_context`는 실제 경로를 기준으로 적어라.
- `semantic_equivalence_group`는 pair, split assignment, dedup envelope에 모두 영향을 준다. 비워 두지 않는 편이 안전하다.
- `canonical_attack_intent_id`, `canonical_untrusted_content`, `mutation_recipe_id`, `reproducibility_seed`는 provenance/reference용이다. 현재 build/run은 이 필드를 직접 쓰지 않는다.

## 6. `split` 필드는 입력이지만 build에서 재배정된다

현재 build는 authoring split를 실행용 split로 보존하지 않는다.

- `_assign_splits()`가 `semantic_equivalence_group` 또는 `template_id` 기준으로 deterministic split를 다시 정한다.
- 현재 샘플 기준으로도 source JSONL의 split 다수가 build package에서 바뀐다.
- 따라서 `split`은 raw catalog 검토용 입력으로 보고, build 후 package split를 실제 실행 기준으로 봐야 한다.

### 이렇게 작성하라

- 초기 의도를 나타내는 split를 적되, build 후 유지된다고 가정하지 말라.
- 같은 `semantic_equivalence_group` 안의 case는 split 재배정 후에도 함께 움직인다고 생각하라.

### 이렇게 쓰지 말라

- “이 케이스는 release heldout에 있어야 한다”는 이유만으로 split 값을 고정 사실처럼 문서화하지 말라.

## 7. KR-EN pair 작성 규칙

현재 구조에는 pair를 두 층으로 나눠 쓰는 편이 안전하다.

### 기본 bilingual pair

- `paired_case_id`를 양방향으로 연결하라.
- pair invariant는 `user_goal`, `protected_asset`, `forbidden_action`, `tool_scope`, `entry_point`, `severity`가 동일해야 통과한다.

### 명시적 pair group

- group-level bilingual tracking이 필요할 때만 `kr_en_pair_id`를 추가하라.
- `kr_en_pair_id`를 쓰면 그 그룹 안에 ko/en이 모두 있어야 한다.
- `paired_case_id`가 있다면 반드시 같은 `kr_en_pair_id` 안을 가리켜야 한다.

### 이렇게 작성하라

- 일반 KR/EN 대응이면 `paired_case_id`부터 맞춰라.
- replay나 structured payload처럼 score/report에서 명시적 pair group이 필요한 경우 `kr_en_pair_id`를 같이 넣어라.

### 이렇게 쓰지 말라

- `kr_en_pair_id`만 넣고 `paired_case_id` backlink를 빼지 말라.
- ko only 또는 en only row에 `kr_en_pair_id`를 달지 말라.

## 8. benign sibling / contrast group 작성 규칙

### `contrast_group_id`

- 공격과 benign control을 같은 실험 묶음으로 보고 싶을 때 사용한다.

### `benign_sibling_id`

- 특정 공격 row가 직접 대응하는 benign row를 가리킬 때 사용한다.
- 현재 validation은 sibling이 반드시 반대 `attack_or_benign`을 가리키게 요구한다.

### `paired_case_role`

- `ko_variant`, `en_control`, `attack`, `benign`, `benign_sibling`처럼 group 안에서 역할을 읽기 쉽게 남기는 필드다.
- 현재 contrast validation은 `paired_case_role`이 `benign...`으로 시작하는 row가 있으면 benign control 존재를 요구한다.

### 이렇게 작성하라

- contrast 실험을 만들면 `contrast_group_id`를 공격/benign 양쪽에 모두 채워라.
- sibling을 두면 두 row가 같은 contrast group에 속하게 맞춰라.

### 이렇게 쓰지 말라

- 공격 row를 benign sibling으로 가리키지 말라.
- benign sibling이 필요한 contrast group인데 benign row를 생략하지 말라.

## 9. replay / tool_input / tool_output / structured payload 작성 규칙

### replay

- `source_stage`는 `replay`로 둔다.
- `source_role`은 현재 validator 기준으로 `memory_note` 또는 `tool_output`이어야 한다.
- `expected_interpretation`은 비워 두지 말라.
- delayed replay라면 `tool_transition_type`, `replay_window`, `delayed_injection_turn`을 같이 채워라.

### tool output / tool input

- `source_stage=tool_output`이면 `source_role=tool_output`이어야 한다.
- tool result가 다음 단계 입력으로 들어가는 경우 `source_stage=tool_input`과 실제 `entry_point`를 맞춰라.
- approval form처럼 사용자가 입력하지만 도구 승인 흐름 일부인 경우, 현재 샘플처럼 `source_stage=tool_input`, `entry_point=form_field`, `source_role=user`, `expected_interpretation=data`를 사용할 수 있다.

### structured payload

- `structured_payload_type`를 비워 두지 말고 현재 샘플 범위의 값으로 적어라.
- 샘플에는 `json`, `function_call`이 존재한다.
- `tool_transition_type`, `threshold_profile`, `normalization_variant`는 structured misuse 비교에 직접 쓰이므로 의도가 있다면 명시하라.

## 10. `analysis_axis`, `primary_mutation`, `register_level`, `expected_guard_stage`를 언제 채우는가

### `analysis_axis`

- 공격이 어떤 실패 축을 겨냥하는지 분명할 때 채운다.
- 현재 taxonomy에 있는 값만 사용하라.
- 값이 불분명하면 억지로 새 이름을 만들지 말고 `other` 계열로 정리하라.

### `primary_mutation`

- 대표 mutation을 하나 고를 수 있을 때 채운다.
- `mutation_family`가 여러 개인 경우 scoring bucket은 `primary_mutation`을 우선 사용한다.

### `register_level`

- 한국어 말투, 높임, 구어체/문어체가 핵심 실험 축일 때 채운다.
- 이 축이 핵심이 아니면 비워 둘 수 있지만, style/register 실험이라면 반드시 채우는 편이 좋다.

### `expected_guard_stage`

- “정상이라면 어디서 막혀야 하는가”를 명시할 수 있을 때 채운다.
- scoring의 `by_guard_stage_alignment`가 이 값을 실제 `failure_stage`와 비교한다.
- replay, normalization, tool policy 시나리오처럼 기대 차단 지점이 분명한 경우는 채우는 편이 좋다.

## 11. 자주 발생하는 오류와 validation 실패 사례

현재 validator가 실제로 잡는 대표 오류는 아래와 같다.

- `paired_case_id` 대상이 없거나 backlink가 끊긴 경우
- pair invariant field가 drift 된 경우
- `kr_en_pair_id`에 ko/en 둘 중 하나만 있는 경우
- `kr_en_pair_id` 밖을 가리키는 `paired_case_id`
- `benign_sibling_id`가 없는 row를 가리키거나 같은 label을 가리키는 경우
- sibling과 본 row의 `contrast_group_id`가 서로 다른 경우
- retrieval/tool_input/tool_output/replay인데 `source_role` 또는 `expected_interpretation`가 비어 있는 경우
- `source_stage=tool_output`인데 `source_role != tool_output`인 경우
- `source_stage=replay`인데 `source_role`이 `memory_note`/`tool_output`이 아닌 경우
- `source_stage=retrieval`인데 retrieval 계열 role이 아닌 경우

추가로 build에서 바로 실패하는 경우도 있다.

- release build인데 `dedup.mode != hybrid_mandatory`
- coverage profile의 `required_values` 또는 `required_combinations`를 못 채운 경우
- 같은 envelope가 dedup 기준에 걸려 case가 drop된 뒤 coverage가 깨진 경우

## 12. authoring checklist

- template와 case의 `template_id`, `attack_family`, `attack_subfamily`가 맞는가
- `<...PLACEHOLDER>` 외의 concrete payload를 넣지 않았는가
- `semantic_equivalence_group`를 의도대로 설정했는가
- KR/EN pair라면 `paired_case_id`가 양방향인가
- `kr_en_pair_id`를 썼다면 ko/en이 모두 존재하는가
- contrast 실험이라면 `contrast_group_id`, `paired_case_role`, `benign_sibling_id`가 일관적인가
- retrieval/tool_input/tool_output/replay라면 `source_role`과 `expected_interpretation`를 채웠는가
- replay/structured payload라면 `tool_transition_type`, `replay_window`, `structured_payload_type` 같은 envelope 필드를 채웠는가
- `analysis_axis`, `primary_mutation`, `expected_guard_stage`가 scoring 목적에 맞게 들어갔는가
- `split` 값을 넣었더라도 build 후 재배정된 결과를 다시 확인했는가
- `pifuzz build`와 `pifuzz validate`를 모두 통과했는가
