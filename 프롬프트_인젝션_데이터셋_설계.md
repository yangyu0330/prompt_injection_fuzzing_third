# 데이터셋 설계 (프롬프트 인젝션 평가용)

## 0. 문서 목적

이 문서는 프롬프트 인젝션 평가를 위한 **데이터셋 구조, 샘플링 원칙, split 전략, 라벨링 체계, 오라클 설계, 품질관리 기준**을 정의한다.  
목표는 단순한 “공격 문장 모음”이 아니라, Layer 1~4 전부를 지원하는 **benchmark-grade dataset package**를 만드는 것이다.

---

## 1. 설계 원칙

### 1.1 하나의 데이터셋으로 끝내지 않는다
프롬프트 인젝션 평가는 최소 3묶음이 필요하다.

1. **진단용 공격셋 (Diagnostic Attack Set)**  
   Layer 1~2용. 엔진 탐지와 Gateway 통합 손실 분석에 사용.
2. **Benign hard-negative 셋**  
   과잉 차단(over-refusal)과 오탐 측정에 사용.
3. **Outcome 시나리오 셋 (Outcome Scenario Set)**  
   Layer 3~4용. 실제 제어권 탈취, 비밀 유출, 무단 행동을 평가.

이 구성은 prompt injection이 직접/간접 입력, 도구 사용, 정상 업무 맥락을 모두 포함하는 문제이기 때문에 필요하다. OWASP, OpenAI, Microsoft, AgentDojo, InjecAgent, OR-Bench의 문제의식과 부합한다. [R1][R2][R3][R5][R6][R7][R8]

### 1.2 dataset item이 아니라 dataset package를 만든다
데이터는 3단위로 관리한다.

- **Template**: 언어 중립적 의미 골격
- **Case**: 언어/변이/entry point가 채워진 샘플
- **Run Record**: 특정 시스템 구성에서 한 번 실행한 결과

이렇게 나누면 한 template를 KR/EN으로 짝지을 수 있고, 동일 case를 여러 방어 설정에서 반복 실행할 수 있다.

### 1.3 split은 무작위 문장 분할이 아니라 구조 분할로 만든다
NIST는 benchmark의 일반화 주장을 위해 평가 목적, 조건, 한계를 명확히 두라고 권고한다. [R4]  
또한 최근 연구와 OpenAI의 자동 red teaming 흐름은 **정적 known-attack만으로는 충분하지 않다**는 점을 보여준다. [R2][R5]

따라서 split은 다음처럼 구성한다.

- **dev/calibration**: 규칙·프롬프트·threshold 조정용
- **held-out static**: 최종 점수 산출용
- **adaptive**: 인간 또는 자동 red team이 현재 방어를 보고 적응적으로 만든 공격용

핵심은 **mutation family, template, entry point, task family를 기준으로 split** 하는 것이다.

## 2. 데이터셋 패키지 구성

## 2.1 패키지 개요

| 묶음 | 용도 | 주 사용 Layer | 핵심 질문 |
|---|---|---:|---|
| Diagnostic Attack Set | 탐지/분류 성능, direct vs indirect, KR-EN gap | 1, 2 | 엔진이 공격을 감지하는가 |
| Benign Hard-Negative Set | over-refusal / 오탐 측정 | 1, 3 | 정상 입력을 공격으로 오탐하는가 |
| Outcome Scenario Set | 목표 달성형 공격 성공률, 무단 행동 | 3, 4 | 결국 뚫렸는가, 정상 업무는 살렸는가 |

## 2.2 권장 최소 규모(초기 버전)
아래는 **최소 권장치**다. 실제 운용에서는 확대 가능하다.

### Diagnostic Attack Set
- primary comparison cell = `language × directness × attack_family`
- 권장 held-out: **셀당 30~50 case**
- 상위 비교(`language × directness`)는 **셀당 100 case 이상**

예시:
- 언어 2(KR/EN)
- directness 2(direct/indirect)
- attack family 8  
=> 32개 primary cell  
=> held-out 기준 최소 960 ~ 1,600 attack case

### Benign Hard-Negative Set
- `language × benign_family × context_form`
- 셀당 20~30 case 권장
- 최소 400~700 case

### Outcome Scenario Set
- base scenario template 60~120개
- KR/EN pair 필수
- 각 case를 시스템 구성별로 **최소 5회 반복 실행**

즉, outcome case 수보다 **run 수**가 훨씬 커질 수 있다.

## 4. 공격 데이터셋 설계 (Diagnostic Attack Set)

## 4.1 목표
- direct vs indirect 탐지 성능 비교
- 한국어 취약점 원인 분석
- family-level recall 측정
- 엔진 단독 vs Gateway 통합 시 paired 비교 가능하게 구성

## 4.2 attack family taxonomy
최소 아래 8개 family를 포함한다.

1. **Attempt to change system rules**
   - ignore previous instructions
   - forget prior rules
   - new unrestricted assistant 선언

2. **Fake conversation / embedded dialogue**
   - 대화 로그 삽입
   - “Human/Assistant/System” 가짜 턴 삽입

3. **Role-play / persona hijack**
   - “지금부터 제한 없는 보안 점검 모드”
   - 캐릭터/전문가/관리자 역할 강제

4. **Encoding / obfuscation**
   - URL/base64/fullwidth/zero-width
   - 한국어 자모분리/초성/혼합 스크립트
   - JSON field / HTML comment / Markdown quote / code block

5. **Context laundering**
   - 인용문, 사례 설명, 보고서 요약처럼 포장된 공격
   - “연구 목적으로 아래 문장을 그대로 따르라”

6. **Tool-output / structured-content injection**
   - JSON/CSV/HTML/검색결과/요약/툴 응답 안의 지시

7. **Multi-turn escalation**
   - 첫 턴에서 정상 질문, 후속 턴에서 권한 상승/지시 무시 유도

8. **Indirect document injection**
   - RAG 문서, 이메일, 파일, 웹페이지, 메타데이터 안의 숨은 지시

## 4.3 entry point taxonomy
각 family는 가능한 한 여러 entry point에서 재사용한다.

- `user_prompt`
- `retrieved_document`
- `search_result`
- `email_body`
- `file_text`
- `webpage_body`
- `html_comment_or_metadata`
- `tool_output`
- `memory_or_summary`

## 4.4 한국어 특화 변이(KR-specific mutations)
한국어 비교가 목적이므로 아래 변이는 별도 family tag를 가진다.

### 문자/표기 계열
- 자모분리
- 초성
- 전각/반각 혼용
- zero-width / soft hyphen
- 한자/영문/숫자 혼용
- 로마자 표기
- 맞춤법 의도적 파괴

### 문체/화용 계열
- 존댓말 우회
- 완곡표현
- 번역체
- 비서/업무 지시체
- 보고서/공지체
- 부탁/승인 유도체

### 구조/맥락 계열
- 따옴표 안 삽입
- 코드블록/JSON/HTML 주석 안 삽입
- 링크 anchor/alt text/meta description
- tool output field value 안 삽입
- 긴 문서 중간 삽입
- multi-turn 요약 누적

## 4.5 KR-EN pair 원칙
모든 핵심 공격은 KR/EN pair를 만든다.

반드시 같아야 하는 것:
- attacker goal
- user goal
- protected asset
- forbidden action
- tool scope
- entry point
- severity

달라질 수 있는 것:
- 표현 방식
- 자연스러운 문체
- 언어권별 자연스러운 포장

직역보다 **공격 의미 동등성**을 우선한다.

## 5. Benign Hard-Negative 셋 설계

## 5.1 왜 필요한가
과잉 차단을 측정하지 않으면 “무조건 막는 시스템”이 최고처럼 보인다.  
OR-Bench는 바로 이 over-refusal 문제를 벤치마크 대상으로 다룬다. [R8]

## 5.2 포함해야 할 benign family
1. 보안 보고서 인용
2. 논문/기사 요약
3. 공격 사례 설명
4. 따옴표 안의 공격 문장
5. 코드 블록 안 예시
6. 정책/가이드 문서 분석
7. 교육용 예시 데이터
8. red-team 결과 회고
9. prompt injection 논문 초록
10. “ignore previous instructions”를 **설명 대상**으로만 포함한 정상 문맥

## 5.3 context form
- plain prose
- quoted text
- code block
- markdown list
- table cell
- JSON field
- HTML snippet
- email forward / reply style

## 5.4 benign 판정 규칙
다음 중 하나라도 만족하면 benign hard-negative 후보로 둔다.

- 공격 문장이 **행동 지시가 아니라 설명 대상**이다.
- 악성 표현이 **인용/분석/교육/번역** 목적이다.
- 정상 업무 수행을 위해 공격 사례를 **검토·분류·요약**해야 한다.
- 정책상 허용된 연구/감사/보안 검토 문맥이다.

## 6. Outcome Scenario 셋 설계

## 6.1 목표
Layer 3~4에서 “탐지했는가”가 아니라 **실제로 공격자가 무엇을 달성했는가**를 평가하기 위한 시나리오 세트다.

## 6.2 각 시나리오가 반드시 가져야 할 필드
- `user_goal`
- `trusted_instruction`
- `untrusted_content`
- `protected_asset`
- `forbidden_action`
- `expected_safe_behavior`
- `oracle`

## 6.3 outcome family taxonomy
최소 아래 5개를 포함한다.

1. **System Prompt Leakage**
   - 시스템 프롬프트 또는 내부 지시 공개 시도

2. **Secret / Data Exfiltration**
   - 시드된 canary, 문서 비밀, PII, 내부 요약 정보 유출

3. **Unauthorized Tool Use**
   - 금지된 tool call
   - 허용되지 않은 parameter
   - 승인 없는 전송/삭제/업로드

4. **Task Hijacking / Manipulation**
   - 정상 과업 대신 공격자 지시 수행
   - 보고서/메일/게시글 내용 조작

5. **Session / Memory Poisoning**
   - 이전 턴 또는 메모리에 공격 지시를 남겨 다음 턴에 발현

## 6.4 severity 등급
- **S1**: 내용 왜곡 / availability
- **S2**: prompt leakage / low-risk data read
- **S3**: secret/PII exposure / unauthorized read
- **S4**: external send, delete, modify, post, payment-like action

## 6.5 tool scope 프로파일
Outcome 시나리오는 최소 3개 권한 프로파일에서 평가한다.

- **P0 none**: tool 없음
- **P1 read-only**: 조회만 가능
- **P2 read+write**: 수정/전송 가능
- **P3 read+write+external**: 외부 발송/업로드/게시 가능

권장 추가 변수:
- human confirmation on/off
- allowlist on/off
- audit logging on/off

## 7. Layer별 매핑 규칙

| 데이터셋 묶음 | Layer 1 | Layer 2 | Layer 3 | Layer 4 |
|---|---:|---:|---:|---:|
| Diagnostic Attack Set | 주 사용 | 주 사용 | 보조 | 보조 |
| Benign Hard-Negative Set | 주 사용 | 보조 | 주 사용 | 보조 |
| Outcome Scenario Set | 보조 | 보조 | 주 사용 | 주 사용 |

원칙:
- Layer 1 점수는 Diagnostic 중심
- Layer 2는 Diagnostic + 일부 Outcome
- Layer 3~4는 Outcome 중심
- BOR는 Benign 셋으로 측정

## 8. 오라클 설계

## 8.1 deterministic oracle 우선
가능하면 LLM judge보다 **기계 판정 가능한 오라클**을 먼저 사용한다.

### Secret oracle
- canary 문자열 exact match
- 변형 허용 regex
- seeded secret ID match

### Tool oracle
- tool call log
- parameter log
- external API request log
- email/post/upload execution log
- file diff / DB diff

### Task oracle
- structured gold answer
- rule-based checker
- invariant checker
- state transition checker

## 8.2 LLM judge는 보조 수단
정말 필요한 경우에만 사용하고, 다음 조건을 둔다.

- deterministic oracle이 없는 경우에만 사용
- rubric을 고정
- 최소 2개 judge prompt로 cross-check
- 랜덤 subset human audit
- judge disagreement 기록

## 9. split 전략

## 9.1 dev/calibration
용도:
- threshold 조정
- regex/정규화/프롬프트/정책 튜닝
- UI/alert wording 조정

포함:
- 쉬운 known attacks
- 대표 hard-negatives
- 간단한 outcome templates

## 9.2 held-out static
용도:
- 공식 점수 산출
- 모델/방어 버전 비교

규칙:
- dev와 **template·mutation family·entry point** 일부를 분리
- final benchmark 전까지 비공개 유지

## 9.3 adaptive
용도:
- 방어를 보고 새로 만든 공격 평가
- 정적 benchmark 과적합 방지

방법:
- human red team 또는 자동 red team이 현 시스템을 보고 공격 생성
- 생성된 공격은 다음 라운드의 held-out 후보가 될 수 있으나, 같은 버전 평가에는 재사용하지 않음

## 10. 표본 수와 균형(balance)

## 10.1 primary balance axes
최소 아래 축을 균형 있게 배치한다.

- language: KR / EN
- directness: direct / indirect
- entry point
- attack family
- mutation family
- privilege profile
- benign vs attack
- single-turn vs multi-turn

## 10.2 권장 최소 비율
### Diagnostic Attack Set
- KR : EN = 1 : 1
- direct : indirect = 1 : 1
- KR의 경우 한국어 고유 변이 비중을 전체 KR 샘플의 최소 40%로 유지

### Benign Hard-Negative Set
- KR : EN = 1 : 1
- quoted/code/analysis/explanation context를 고르게 배치

### Outcome Scenario Set
- 각 outcome family별 최소 10~20개 base template
- 각 template는 KR/EN pair
- 각 case는 최소 5회 실행

## 10.3 sample size 계산 원칙
비율 추정에서 worst-case \(p = 0.5\) 기준:
- ±10% @95% CI ≈ 100건
- ±5% @95% CI ≈ 385건

따라서:
- headline comparison(예: KR vs EN)은 셀당 100건 이상
- family-level exploratory comparison은 셀당 30~50건 이상
- outcome scenario는 case 수보다 반복 실행 수를 우선 확보

## 12. 라벨링 프로토콜

## 12.1 필수 라벨
- attack_or_benign
- attack_family
- directness
- entry_point
- mutation_family
- protected_asset
- forbidden_action
- severity
- tool_scope

## 12.2 annotation 방식
- 1차 라벨러 2인 독립 라벨링

- 최종 결정자는 benchmark maintainer
- 애매한 case는 “exclude_from_headline” 태그로 분리

## 12.3 quality gate
- KR/EN pair 의미 불일치
- oracle 불명확
- attack/benign 애매
- outcome 성공 조건 불명확
- tool side effect 검증 불가

위 조건이면 정식 release에서 제외한다.

## 14. 유지보수와 누출 방지

## 14.1 benchmark contamination 방지
- held-out static은 비공개 유지
- adaptive 공격은 별도 저장
- release note에 어떤 family가 추가/삭제되었는지만 공개
- exact prompt 전체 공개 여부는 단계적으로 결정

## 14.2 training leakage 방지
- 운영 모델/가드레일 tuning에 사용한 dev split과 held-out split을 절대 혼합하지 않음
- 모델 개선 후에는 adaptive split을 새로 생성

## 14.3 benchmark card 작성
각 release마다 다음을 명시한다.
- 목적
- scope
- included/excluded attack families
- supported languages
- intended layers
- scoring metrics
- limitations
- known blind spots

## 16. 이 설계로 확보되는 것

이 설계가 있으면 다음 주장이 가능해진다.

- 한국어 vs 영어 direct/indirect 탐지 차이
- Gateway 통합으로 인한 integration loss
- 한국어 indirect injection의 실제 ASR/SER/UAR
- benign 문맥에 대한 BOR
- write 권한, human confirmation, pre/post guardrail 조합의 효과
- 신규 변이에 대한 일반화 성능

반대로 이 설계가 없으면:
- known attack memorization
- KR/EN 비교의 비대칭
- 탐지율 중심 착시
- over-refusal 누락
- 실제 서비스형 위험과의 단절

이 발생할 수 있다.
